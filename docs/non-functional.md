# 非機能要件 — chalo

機能の正しさ以外で満たすべき品質・運用・公開の要件。

## 対応範囲 [確定]

- プラットフォーム：iOS のみ。
- 言語：日本語のみ。
- 外観：ライトモードのみ（ダークモード非対応）。
- 収益：無料を前提に設計。
- 対応下限 iOS：Expo SDK と利用ライブラリの既定に従う（恣意的に上げ下げしない）。
- 対応端末：iPhone のみ（iPad 非対応）。

## オフライン / 同期 [確定]

- 同期は**起動時・フォアグラウンド復帰時**に行う（リアルタイム常時同期はしない）。`adr/0004`。
- オフライン時は**アプリを使えない状態にする**（`adr/0008`）。
  - 全画面の**「オフラインです」表示**に切り替え、インターネットの復旧を促す。復旧してから戻ってくるよう案内する。
  - 閲覧・作成・編集を含め、すべての操作を止める（読み取り専用のキャッシュ表示もしない）。
- 編集衝突は編集ロックで防ぐ（`adr/0005`）。データ更新の基本は last-write-wins。

## エラーハンドリング方針 [確定]

表現の使い分け：
- **一時的・復帰可能** → トースト＋リトライ。
- **破壊的・要判断** → ダイアログ。
- **フォームの不備** → インライン表示。
- **サーバ側 push の送信失敗** → ユーザーには見せない（サーバでログ／リトライ）。
- **ログ書き込み失敗** → 握りつぶす（アプリ動作に影響させない）。

### 想定エラー網羅リスト

| 事象 | 挙動 |
|---|---|
| 起動時・操作中にオフライン | 全画面「オフラインです」表示で全操作を停止。復旧を促し、復旧後に通常画面へ戻す。OIDCログインもネット必須 |
| 起動時同期の取得失敗（オンラインだがエラー） | キャッシュがあれば出しつつ「再取得」、無ければエラー＋再試行 |
| 保存失敗（作成/編集/おしまい/削除） | トースト＋リトライ。楽観更新時はUIロールバック |
| 編集ロック衝突 | 相手が編集中のプランは編集ロック画面（F-9。「〇〇 が編集しています」）を表示し操作をブロック（内容の閲覧のみ） |
| セッション切れ／認証エラー | 再ログイン導線 |
| 通知権限の拒否 | 機能縮退＋設定の通知オンボタン→iOS設定へ |
| 写真権限の拒否／限定アクセスで対象日が見えない | アルバム空表示＋権限拡大の導線 |
| カレンダー権限の拒否 | 機能縮退＋設定のカレンダー許可→iOS設定へ |
| カレンダー追加／更新／削除の失敗 | トースト＋リトライ |
| 外部でカレンダーイベントが手動削除されリンク切れ | リンク解除して「未追加」へ |
| 招待コードが無効／期限切れ／既にペア済み／自分のコード | それぞれのエラー表示 |
| 期限通知の予約失敗（ローカル） | 静かに記録、致命としない |
| 振り返り／作成通知のサーバ送信失敗 | ユーザーに出さず、サーバでログ／リトライ |
| 通知タップ先のプランが削除済み | 「見つかりません」 |
| 場所検索の失敗 | トースト＋リトライ |
| ログ書き込み失敗 | 握りつぶす |

## 通知（非機能面） [確定]

- 許可は必要になる文脈で取得（JIT＋プライミング）。通知は**ペア成立直後**に要求する。`domain/onboarding.md`。
- ユーザーごとのタイムゾーンを保持し、配信時刻の算出に使う（各通知の配信時刻は `domain/notifications.md`）。
- 2系統：端末ローカル予約（期限）＋サーバ push（作成・振り返り）。`adr/0007`。

## ログ / 可観測性 [確定]

- ログは**端末内に保存**（形式は **NDJSON**）。書き込み負荷を抑える設計（バッファ・フラッシュ契機・ローテーション基準）は `adr/0011` を正とする。
- 設定の「不具合報告 / ログ送信」で端末内ログを送付。送信時に**任意でコメント**を添えられる。送られたログは `bug_reports` に1行として入り、**アカウント削除まで保持**（`data-model.md`）。
- クラッシュ／エラー監視（Sentry 等）を入れるかは要確定 [保留]。入れる場合もプライバシーに配慮。

## パフォーマンス [提案]

- 「手早く貯める」が要。プラン作成は楽観更新で即時にリスト反映する。
- アルバムは対象日のアセットだけを日付クエリで取得し、サムネイルを遅延ロード。全件走査しない。
- 一覧はページング／仮想化を検討（プランが増えても軽快に）。

## セキュリティ / プライバシー [確定]

- 認証は Supabase OIDC（Google / Apple）。
- データアクセスは RLS で同一ペア（またはソロ本人）に限定（`data-model.md`）。
- アルバム写真は端末外に出さない。
- アカウント削除はアプリ内で完結し、退会者の個人情報を削除（`domain/pairing.md`）。

## App Store 公開要件 [確定]

- **Apple サインイン提供**（他社ソーシャルログインを提供する場合の審査要件）。Google と併せて満たす。
- **アプリ内で完結するアカウント削除**（必須）。
- **プライバシーポリシー / 利用規約**（アカウント＋写真アクセス＋push があるため必須）。設定からアクセスできる。
- **写真ライブラリ利用目的の文言（Info.plist `NSPhotoLibraryUsageDescription` 等）**。限定アクセス時の挙動も実装。
- **通知・カレンダーの利用目的文言**（`NSUserNotifications` 系 / `NSCalendarsUsageDescription`）。
- **App Store メタ情報**：カテゴリ、年齢レーティング、プライバシーラベル（収集データの申告）。
- 写真・カレンダー・通知の各権限は、必要になる文脈が明確であること。

## アクセシビリティ [確定]

- Dynamic Type（文字サイズ追従）・VoiceOver（スクリーンリーダー）には対応しない。
