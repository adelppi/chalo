# 未確定論点・決定バックログ — chalo

各ファイルに散らばる `[保留]` / `確定させたい論点` を1か所に集約したハブ。
ここで扱うのは**機能の有無ではなく、値・形式・方式といった設計の詳細**。機能を持つ／持たないは `features.md` で確定済み。

## 使い方

- 各論点には ID（`Q-<領域>-<番号>`）を振る。各文書の本文からはこの ID で参照してよい。
- 決まったら結論を**該当する元ファイルへ反映**し、ここは末尾の「決定済み（記録）」へ1行で落とす（ハブと本文の二重管理にしない＝ここは索引、正は各文書）。
- 優先度は付けず、領域別に並べる。

**状態**：`未着手` / `議論中` / `暫定案あり` / `保留` / `提案`（決定済みは末尾の記録へ）

## オンボーディング（Q-ONB）

### Q-ONB-1
**起動時 pairState 取得失敗（オフライン等）時のフォールバック**
- 内容：起動ゲートはオンボーディングの要否を決める前に、サーバの pairState（`profiles.pair_id` の有無）の取得を待つ（Issue #56）。取得が失敗・停止（オフライン等）した場合、(a) ローカル進捗に従ってオンボーディングを表示する（＝pairState 抜きで判定）か、(b) 前回取得できたペア状態を端末に保持しておき、それを使うか。
- 暫定案：(a) を実装。既にペア済みだがローカル完了フラグが無い端末（招待コード成立前の旧バージョンでペアが成立した端末・再インストール等）がオフラインで起動すると、まれにオンボーディングが再表示されうる（オンライン復帰後の起動では発生しない）。
- 関連：`domain/onboarding.md`（起動時の着地先・優先順位1）
- 状態：暫定案あり
- 結論：
