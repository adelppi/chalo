# ADR-0001: バックエンドに Supabase を採用する

- ステータス: 採用 [確定]
- 関連: data-model.md, adr/0004, adr/0007

## コンテキスト

認証・データベース・ペア間同期・プッシュ送信のサーバ処理が必要。React Native + Expo、かつAI駆動開発を前提に、スキーマやポリシーをAIに書かせやすく、運用負荷の低い基盤が欲しい。

## 決定

**Supabase** を採用する。

- 認証：Supabase Auth の **OIDC（Google / Apple サインイン）**。
- DB：Postgres。RLS で同一ペア／ソロ本人にアクセスを限定。
- 同期：起動/復帰時の取得（Realtime 常時接続は使わない。`adr/0004`）。
- サーバ処理：Edge Function ＋ **pg_cron**（振り返り通知の日次バッチ。`adr/0007`）。
- ストレージ：**ほぼ使わない**。アルバムは端末ローカル完結のため、写真をクラウドに置かない。

## 結果

- 良い点：Postgres＋RLSで権限を宣言的に書ける／AIにスキーマ・ポリシー・SQLを書かせやすい／無料枠で開始できる／pg_cronで定期実行が完結。
- 留意点：RLS 設計を誤るとデータ漏れに直結するため、ペア境界のポリシーは重点的にレビューする。Realtimeを後から足す余地は残す。

## 検討した代替案

- **Firebase**：実績豊富だが、ペア境界の権限表現とリレーショナルなプラン管理はPostgres＋RLSの方が素直。
- **自前バックエンド**：本アプリの規模・速度に対して過剰。
