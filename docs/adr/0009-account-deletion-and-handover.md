# ADR-0009: アカウント削除と退会時のデータ取り扱い

- ステータス: 採用 [確定]
- 関連: domain/pairing.md, non-functional.md, data-model.md

## コンテキスト

ペア解除はアカウント削除と連動する。退会者の個人情報削除は法令・App Store の要件。「相手が消すと共有データも消えて、残された側が読めない・書き出せない」という鶏卵問題を避けつつ、設計はできるだけ単純に保ちたい。chalo は二人で使うアプリで、**一人での継続利用は想定しない**（ソロモードは招待までの試用・事前登録期間）。

## 決定

- アカウント削除は**アプリ内で完結**する。
- **譲渡（Handover）・所有権モデルは持たない。** 残った側をソロに戻して使い続けさせる導線も作らない。
- パートナー（A）が削除したとき：
  - Aの個人データ（プロフィール、認証連携、push token、Aが発行した招待コード）を**削除**する。
  - 共有プランは**消さず保持**する。Aを指す参照は付け替え・削除する：
    - `plans.owner_id`（所有者＝作成者）→ **B に付け替え**（NOT NULL 維持）。
    - `plans.locked_by` → **null**。
    - `push_tokens` / `invites`（Aのもの）→ **削除**（他人に付け替えない）。
  - 各FKに `ON DELETE`（CASCADE / SET NULL）を定義し、付け替えと合わせて、Aの削除がDB制約で失敗しないようにする。削除はサーバ側の関数（service role）で1トランザクションにまとめる。
  - Aが作成したプランは、**メモ末尾に元の作成者を追記**して残す（表示名を実名で残す。文言は `domain/pairing.md`）。
- 残った側（B）は**ロック状態**になり、「全プランの書き出し」か「アカウント削除」しかできない。通常利用・新規招待の導線は持たない。
- Bがアカウントを削除すると、残っていた共有プランと `pairs` 行もカスケード削除され、A・B のデータが完全に消える。
- **Bが動くまで共有データはDBに残る（ブロッカー方式）。** 放置時の自動削除（保持期限）は設けない。
- 書き出しは退会時の救済ではなく**設定の常設バックアップ**。全プランを連結した1つのテキスト（.txt）を生成し、iOS共有シートで共有する（ファイル保存 / AirDrop 等はOS任せ）。写真は対象外。
- アルバムは各自の端末ローカル写真で、退会・削除の対象外。

## 結果

- 良い点：設計が最小（譲渡・ラベル列・所有権付け替えモデルが不要）。鶏卵問題は書き出しで解消。退会者の個人・端末データは確実に消える。
- 割り切り：
  - Bは chalo を一人で使い続けられない（書き出して畳む or 削除）。二人で使うアプリという前提に沿う。
  - 付け替えで作成者表示はBになるが、元の作成者名はメモに残す（＝退会者の実名が相手の書き出しに残ることを許容）。
  - Bが放置すると共有データがDBに残り続ける（保持期限を設けないため）。

## 却下した案

- **譲渡（旧案）**：`owner_id` を残った側に付け替え、ソロデータとして継続利用可能にする。残った側をソロに戻す・再ペアできるようにするための状態管理（`profiles.pair_id` クリア、`pairs` 削除、`creator_label` 置換）が増える。一人使いを想定しない方針と単純さを優先して不採用。
- **自動パージ（保持期限）**：一定期間後に放置データを自動削除する案。DB残留を確実に消せるが、スケジュールジョブが増える。単純さを優先して不採用（ブロッカー方式を許容）。
