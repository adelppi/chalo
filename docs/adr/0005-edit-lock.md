# ADR-0005: 編集ロックで競合を防ぐ（TTLあり）

- ステータス: 採用 [確定]
- 関連: adr/0004, data-model.md, domain/plan-lifecycle.md

## コンテキスト

二人が同じプランを同時に編集する競合を防ぎたい。一方でリアルタイム同期は使わない方針（`adr/0004`）。常時購読なしで「相手が今編集中か」を知る必要がある。

## 決定

プラン行に `locked_by` と `locked_at` を持ち、次の手順でロックする。

1. プラン詳細を開いた時、および編集ボタンを押した瞬間に、**その1件だけ最新取得**してロック状態を確認する。
2. 相手がロック中（かつTTL内）なら**編集ロック画面**（F-9。「**〇〇 が編集しています**」）を表示し、編集・削除・おしまい等の操作をブロックする。プラン内容（タイトル・日にち・メモ）の閲覧のみできる。
3. 空いていれば自分のロックを立てて編集開始。
4. 保存／キャンセルでロック解除。

- **TTL（自動失効）**：例 5分。アプリが落ちてロックが残っても、TTL経過で自動的に空く。
- **ほぼ同時押しの競合**：考慮しない（発生確率が低く、許容する）。

## 結果

- 良い点：リアルタイム同期なしで競合を防げる／実装が局所的（編集開始時の1件取得のみ）。
- 留意点：TTLの値は体験とのトレードオフ（短すぎると編集中に失効、長すぎるとクラッシュ後に待たされる）。5分を初期値に調整。
- 「ほぼ同時押し」を許容したため、極稀に二人が同時にロックを立てる可能性が残る（last-write-wins で最終的整合）。

## 検討した代替案

- **条件付き更新で同時押しも厳密に弾く**：堅いが、発生確率に対して複雑。今回は不要と判断。
- **ロックせず last-write-wins のみ**：競合時に相手の編集が黙って消えるため、UX上ロックを採用。
