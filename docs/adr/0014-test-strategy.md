# ADR-0014: テスト戦略

- ステータス: 採用 [提案→要承認]
- 関連: adr/0001, adr/0006, non-functional.md

## コンテキスト

AI駆動開発で実装が速く動く分、回帰を機械的に検知したい。一方で iOS シミュレータ越しの重いテストを増やすと反復が鈍る。ロジックの単体検証と、ユーザー体験の通し確認の2層に絞る。

## 決定

採用するテストは次の2つ。

| テスト | 粒度 | ツール |
|---|---|---|
| ロジックテスト | 純粋関数・ユーティリティのみ（副作用なし。日付判定・通知スケジュール計算・タイムゾーン変換・マージなど）。UI・フックは対象外 | Jest（`jest-expo`） |
| E2E テスト | 機能ごとに作成し、各機能で王道のユーザーシナリオを網羅する。正常系・異常系の両方を含める | Maestro |

- **ロジックテストは純粋関数だけ**に絞る。副作用のない関数に切り出せる計算・判定を対象とし、UI 描画やフックは E2E に委ねる。安定して高速、保守コストが低い。
- **E2E は機能単位**で構成する。1機能につき、その機能の王道の使い方を通す正常系と、権限拒否・オフライン・入力不備などの異常系を両方カバーする。
- **E2E は実行の動画を記録して格納する**。Maestro の録画を成果物として保存し、テスト失敗時に画面の動きから原因をたどれるようにする。

## 対象外

- **ペア境界（RLS）の自動テストは持たない**。ペア境界のポリシーはレビューで担保する（`adr/0001`）。
- **クラッシュ／エラー監視（Sentry 等）はスコープ外**。本番での回帰検知は別に扱う（`non-functional.md` の保留事項）。

## 結果

- 良い点：純粋ロジックを単体で固めつつ、体験の通しは機能ごとの E2E で押さえる。対象がツールと1対1に対応し、AIに「どこに何を書くか」を指示しやすい。
- 留意点：純粋関数に切り出せないロジックは E2E 側で担保することになる。ロジックを純粋関数へ寄せる設計を心がける。
- 留意点：機能ごとに正常系・異常系を網羅すると E2E 本数が増える。実行時間が反復を損なわないよう、CI での並列実行・対象の見直しを継続する。

## 検討した代替案

- **コンポーネントテスト（React Native Testing Library）を挟む**：描画・操作を単体で検証できるが、E2E と守備範囲が重なり保守が二重になる。今回はロジック（純粋関数）と E2E の2層に絞る。
- **E2E を主要な数本に絞る**：軽いが、機能ごとの異常系まで手動 QA に委ねることになる。機能単位での網羅を優先する。
