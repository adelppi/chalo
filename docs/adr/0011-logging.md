# ADR-0011: ログは端末内・低負荷

- ステータス: 採用 [確定]
- 関連: non-functional.md, data-model.md

## コンテキスト

不具合報告のためにログが欲しいが、書き込み負荷でアプリ体験を損ねたくない。プライバシー上、ログは端末内に留め、送るかどうかはユーザーが決める。

## 決定

### 端末内の記録

- ログは **端末内に保存**する（クラウドへ常時送信しない）。
- 形式は **NDJSON**（改行区切りで1行に1つの JSON オブジェクト。時刻・レベル・画面などを構造化して持つ）。追記専用のテキストなので1行足すだけで書け、行単位でローテーションできる。
- 低負荷方式：
  - **メモリ上のリングバッファ**に溜める。
  - **バックグラウンド移行時**にまとめて1ファイルへ**追記**する。ただし **error 行は即フラッシュ**する（クラッシュ直前の重要ログを取りこぼさない。error は稀なので書き込み負荷はほぼ増えない）。
  - 保持は **30日 かつ サイズ上限**、先に当たった方で古い行から破棄する。
- **ログ書き込み失敗は握りつぶす**（アプリ動作に影響させない）。

### 送信とサーバ保持

- 設定の「不具合報告 / ログ送信」を押したときだけ送る。押すときに **任意でコメント**（症状）を添えられる。
- 送信1回が **`bug_reports` の1行**（ログ1行 = 1レコードにはしない）。端末にあった分（最大30日）を `logs` 列に**生の NDJSON テキストのまま**格納する（`data-model.md`）。
- 受信レポートは**アカウント削除まで保持**（自動パージなし）。本人の削除時に `ON DELETE CASCADE` で消える。

### ログに含めない情報

**ユーザーが入力した内容は一切書かない（内容ゼロ方式）。**

- 含めない：プラン本文（タイトル・メモ・場所名・URL）、表示名・相手の呼び方、招待コード、認証情報・トークン類（push token 含む）、位置座標、写真の識別子・パス。
- 含めてよい：画面名・操作イベント名、各種 ID（UUID）、エラー種別・スタックトレース、アプリ／OS バージョン・端末モデル、時刻。

## 結果

- 良い点：書き込み頻度が低く負荷が小さい／プライバシーに配慮（送るのはユーザーが選んだ時だけ）／error は即フラッシュでクラッシュ直前も追える。
- 留意点：ログが DB に入るため、含めない情報の線引きを上記「ログに含めない情報」の通り固定する。
