# ドメイン: 通知

chalo の通知は、新しい予定を煽らない。
プランを作成したとき／すでに貯めたものが終わる前／また行ける頃に知らせるだけ。3種類ある。

この文書は通知の**振る舞い（種類・条件・タイミング）**を定義する。どの方式（サーバ push／端末ローカル予約／日次バッチ）で実現するかは `adr/0007-notifications-architecture.md` を正とする。許可の取得方針は `non-functional.md` を正とする。どの通知も**タップでそのプランへ遷移**する。

---

## 1. 作成通知（Creation push） [確定]

パートナーがプランを作成したことを知らせる。

- **トリガー**：パートナーがプランを作成したとき。
- **頻度**：作成のたびに毎回。束ねない（短時間の連続作成も都度送る）。
- **対象**：パートナー（作成者本人には送らない）。
- **遷移先**：作成されたプランの詳細。
- **本文に出す作成者の名前**：**受信者から見た**相手の呼び方（規則は `domain/pairing.md`「相手の名前の表示」）。
- **対象外**：編集・日付変更・おしまい等は通知しない（通知するのは作成のときだけ）。

---

## 2. 期限通知（Deadline push） [確定]

「これを過ぎると行けなくなる」プランを、終わる前に知らせる。通知の主役。

- **トリガー**：プランに**期限**が入っていること。期限を入れること自体が「通知してほしい」の意思表示。
- **タイミング**：**期限の2週間前の朝9:00に1回だけ**（端末ローカル予約のため端末のタイムゾーンで鳴る）。
- **送らない条件**：
  - すでに**日付（行く予定日）が入っている**場合（もう行く気なので不要）。
  - 作成（または期限設定）時点で**期限まで2週間を切っている**場合（そもそも送らない）。
- **対象**：二人とも。
- **遷移先**：該当プランの詳細。
- **オフラインでも鳴る**：固定オフセット（2週間前）の通知のため、オフラインでも届く。

### 期限通知のライフサイクル
- 期限を追加 → 2週間前に予約。
- 期限を変更 → 旧予約を消して再予約。
- 期限を削除 → 予約を消す。
- 日付（予定日）が入った → 予約を消す（送らない条件に合致）。
- 手動おしまい → 予約を消す（もう閉じたプランを促さない）。自動おしまいは日付のあるプランにしか起きず、日付が入った時点で予約は消えているため、対応不要。
- プラン削除 → 予約を消す。

## 3. 振り返り通知（Reminiscence push） [確定]

1年前に行った場所を、また行ける頃に思い出させる。

- **対象**：**1年前におしまいになったプラン**。
  - 「おしまい」＝予定時刻を満了して自動で閉じた、または手動でおしまいボタンを押して閉じたプラン。
  - これにより**日付を設定しなかったプランも振り返れる**（おしまい日が基準になるため）。
- **何年前まで**：**1年前のみ**（2年・3年前は送らない）。
- **タイミング**：1年前と同じ日に知らされても、いきなりは動けない。そこで**「1年前のおしまい日」の1年後の、2週間前**に通知する（また計画できる猶予を持たせる）。
  - 式：通知日 = （おしまい日 + 1年） − 2週間。配信時刻は**朝9:00**（`profiles.timezone` 基準）。
- **頻度の上限**：1日1通まで。同じ日に複数該当する場合の扱いは下記。
- **遷移先**：該当プランの詳細（そこからアルバムが見られる）。「もう一度プランにする」ボタンは不要。

### 同日に複数該当した場合 [提案]
- タップ先が1プランに定まる必要があるため、**プランごとに別々の push**を基本とする。
- 「1日1通まで」の上限と矛盾しないよう運用を確定させたい論点：複数該当時に「まとめて1通（タップでリスト）」にするか、「プランごとに送る（同じ朝に複数届くのを許容）」か。頻度が低いので後者を許容する案を推奨。要確定。

## 横断仕様

### 通知許可
- 許可の取得・拒否時の縮退方針は `non-functional.md`（権限）を正とする。拒否時は機能を縮退し、設定の通知オンボタン → iOS 設定へ導く。

### 配信時刻とタイムゾーン [確定]
- **作成通知は即時**（配信時刻の概念なし）。**期限・振り返り通知は朝9:00**に届ける。
- 期限通知は端末ローカル予約のため端末のタイムゾーンで解決する。振り返り通知はサーバ送信のため、**ユーザーごとに保存したタイムゾーン**（`profiles.timezone`）で算出する。

### 送信失敗 [確定]
- サーバ push の送信失敗は**ユーザーには見せない**。サーバ側でログ・リトライ（`non-functional.md`）。

### 通知先プランが削除済み [確定]
- タップ時に対象プランが無ければ「見つかりません」を表示。

## 実装方式

各通知をどの方式（サーバ push／端末ローカル予約／日次バッチ）で実現するかは `adr/0007-notifications-architecture.md` を参照。
