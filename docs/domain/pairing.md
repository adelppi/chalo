# ドメイン: ペアリングと退会時のデータ取り扱い

二人をどう繋ぎ、どう解くか。カップルアプリで最も繊細な領域。

## 基本モデル [確定]

- chalo は **1対1のペア**を前提とする。
- ユーザーは **ペアが成立する前から一人で使える**（ソロ利用）。プランを貯められる。
- 認証は Google / Apple サインイン。実現方式は `adr/0001-backend-supabase.md`。

## ペアの成立（招待コード） [確定]

方式は**招待コードの手入力**で確定（招待リンクは持たない）。コードは**6桁の数字**、有効期限は**発行から24時間**。

1. ユーザーAが招待コードを発行する（設定画面のペア欄）。
2. ユーザーBがコードを入力する。
3. 1対1のペアが成立し、以降プランが共有される。

招待は**コードの手入力**で行う。ディープリンク（招待リンク）は持たない（`domain/onboarding.md`）。

### コードのルール [確定]
- 形式：**6桁の数字**（ランダム生成。衝突時は再生成）。
- 有効期限：**発行から24時間**。1コード1回使用。
- 再発行はいつでもできる。**再発行すると旧コードは失効**する（有効な未使用コードは常に1つ）。

### ソロプランの合流 [確定]
- ペア成立時、**両者がソロで貯めていたプランはすべて自動で共有プールへ合流**する。選択UIは設けない（プランは全共有の思想通り。「選ばせる」「合流しない」はどちらも相手に見えないプランを生むため持たない）。
- 成立処理（コードの使用確定・ペア作成・両者のペア所属・ソロプランの合流）は**サーバ側の関数で1トランザクション**として実行し、半端な状態を残さない（`data-model.md`）。

### 招待まわりのエラー [確定]
無効／期限切れ／既にペア済み／自分のコード、の各ケースでそれぞれ分かりやすいエラーを出す（`non-functional.md`）。

## プランの共有範囲 [確定]

- ペア成立後、プランは**全共有**（サプライズ・隠しプランは無い）。
- 片方が作成・編集すると、同期タイミングでもう片方に反映される（`adr/0004-sync-strategy.md`）。

## ペア解除と退会

### 基本方針 [確定]
- **ペア解除はアカウント削除と連動する**。独立した「ペアだけ解除」操作は持たない。
- アカウント削除はアプリ内で完結する（App Store 要件。`non-functional.md`）。
- chalo は二人で使うアプリ。**一人での継続利用は想定しない**。ソロモードは「ペアを招待するまでの試用・プラン事前登録の準備期間」という位置づけ（`domain/onboarding.md`）。

### パートナー（A）が削除したとき [確定]
Aがアカウントを削除すると、次のとおり処理する。「相手が消すと共有データも消えて、残された側が読めない・書き出せない」という鶏卵問題は、共有プランを残して残った側に書き出させることで避ける。**譲渡・所有権モデルは持たない。**

- **Aの個人データは削除**する（プロフィール、認証連携、push token、Aが発行した招待コード）。
- **共有プランは消さない**。残った側（B）が書き出せるように保持する。
- Aを指す参照は、Aの削除がDB制約で失敗しないよう付け替え・削除する（各FKに `ON DELETE` を定義。`data-model.md`）：
  - `plans.owner_id`（所有者＝作成者）→ **B に付け替え**（NOT NULL のまま保つ）。
  - `plans.locked_by`（Aが保持中のロック）→ **null にクリア**。
  - `push_tokens` / `invites`（Aのもの）→ **削除**。Bには付け替えない（端末トークンを他人に回すと通知の誤送信になる）。
- Aが作成したプラン（付け替え前の `owner_id = A`）は、**メモ末尾に元の作成者を追記**する。
  - 例：既存メモ＋改行＋「（このプランは A（削除済み）が作成）」。
  - 付け替えで作成者表示がすべてB（＝自分）になるため、元の作成者をテキストとして残す。
  - Aの**表示名を実名で残す**（退会後も相手の書き出しに名前が残ることを許容する方針）。
  - 追記対象はAが作ったプランだけ（Bが作ったものには付けない）。

### 残った側（B）の状態 [確定]
- Bがアプリを開くと、パートナー消失を検知して**ロック状態**に入る。
- ロック中にできるのは2つだけ：
  1. **全プランの書き出し**（下記。設定の常設バックアップと同一実装）。
  2. **アカウント削除**。
- 通常の閲覧・編集・新規作成・新しい招待はできない。chalo を一人で使い続ける導線は用意しない。

### ロック画面の文面 [確定]

検知時に全画面で表示する。事実を簡潔に伝え、飾らない。

> **パートナーがアカウントを削除しました**
>
> ふたりのプランの閲覧・編集はできなくなりました。
> プランはテキストファイルに書き出して残せます。
>
> ［プランを書き出す］／［アカウントを削除する］

### 全データの消去 [確定]
- Bがアカウントを削除すると、**Bの個人データと残っていた共有プラン（＋`pairs` 行）をまとめてカスケード削除**する。この時点で A・B 由来のデータがすべて消える。
- 書き出しはコピーを外に出すだけで、DBのデータは消さない。データをDBから消す手段は「アカウント削除」のみ。
- **Bが動くまで共有データはDBに残る（ブロッカー方式）。** 放置された場合の自動削除（保持期限）は設けない。

### アルバム [確定]
- アルバムは各自の端末ローカル写真。退会・削除の対象外で、各自の端末にそのまま残る。

## 書き出し / エクスポート [確定]

- **設定からいつでも実行できる常設バックアップ**として用意する（退会時だけの救済ではない）。パートナー削除後のロック状態でも同じ実装を使う。
- 全プランの内容を連結した**1つのテキストファイル（.txt）**を生成し、**iOSの共有シート**に渡す（ファイルに保存 / AirDrop / メール等はOSに委ねる）。
- **アルバム写真は対象外**（端末ローカルにあるため）。

### 書き出しに含める項目（目安） [提案]
タイトル、ステータス、日付・時刻、期限、場所（名称・座標）、参考URL、メモ、作成者、おしまい日、作成日時。

## オフライン時 [確定]
- オフライン時はアプリ全体が使えない（全画面「オフラインです」表示、`adr/0008-offline-policy.md`）。ペアリング操作（招待コード発行・入力）を含め、すべてオンライン復帰後に行う。
