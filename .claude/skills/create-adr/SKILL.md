テーマ: $ARGUMENTS

## 手順

1. **採番**: `docs/adr/` の既存ファイルの最大番号 + 1(4桁ゼロ埋め)。ファイル名は `XXXX-英語スラッグ.md`。
2. **形式**: 既存 ADR(例: `docs/adr/0015-directory-structure.md`)を1つ開いて文体・構成を確認し、同じ形式で書く。

   ```markdown
   # ADR-XXXX: タイトル

   - ステータス: 採用 [提案→要承認]
   - 関連: adr/XXXX, docs/…

   ## コンテキスト
   なぜこの決定が必要か。

   ## 決定
   何をどうするか(具体的に)。

   ## 結果
   良い点・留意点。

   ## 検討した代替案
   各案と、採用しない理由。
   ```

   ステータスは `採用 [提案→要承認]` で作成し、ユーザーが承認するまで `[確定]` にしない。
3. **内容の充実**: 選択肢が会話で十分に議論されていなければ、主要な代替案を調査・比較してから書く。「検討した代替案」が空の ADR は作らない。
4. **既存との整合**: 既存 ADR と矛盾しないか確認する。既存の決定を覆す場合は、旧 ADR のステータスを「廃止(superseded by ADR-XXXX)」に更新し、新 ADR の `関連:` に旧番号を書く。
5. **目次更新**: `docs/README.md` の構成ツリーに追記する。決定が「一次情報の所在」表に関わるなら、その表も更新する。
6. **open-questions.md**: この ADR が `docs/open-questions.md` の項目を解決するなら、該当項目の扱い(削除・解決済みマーク)をユーザーに確認する。

すべて日本語・である調(既存 ADR に合わせる)。
